# 评估协议：Judge 流程、Gate、GSB、指标与风险控制

## 〇、评估对象与模式

评估对象只有产出的成片。运行前先确定模式：

- **有参照（with-reference）**：视频 + Brief + 产品事实 + 平台/品牌要求，六层全评；
- **无参照（no-reference）**：只有视频，仅评 dimensions.md 中不需参照的维度，报告必须注明降级范围。

无论哪种模式，**客观检查轨先行**：证据提取阶段同时运行确定性工具检查（ffprobe 规格、OCR 错字比对、ASR 语言检测、LUFS 响度、帧间差分），结果写入 `evidence.objective_metrics`。L2/L3 打分前另跑**密集抽帧人工复核台账**（≥0.5fps 全片拼图 + 可疑帧放大，产物 `objective/frame_review_ledger.json`，记录带时间戳的确认画面 bug 与单屏停留时长）——2026-07-17 实测双 Judge 模型在稀疏抽帧下系统性漏报布局级 bug（models.md 坑 #11），台账作为确认问题清单注入 judge，冲突时以台账为准。客观轨能给出结论的项不再交给 Judge，客观轨与 Judge 冲突时以客观轨为准（见 evolution.md）。

Judge 模型采用双模型策略（gemini-3.5-flash 全维度 + doubao-seed-2-pro 视觉专项），分工、合并规则、评分虚高校准与已知坑见 [models.md](models.md)。

## 一、Judge 四步流程

### 第一步：观察视频（Evidence Extraction）

把视频转化为模型可理解的证据包（见 `schemas/evidence.schema.json`）：

- 关键帧（全视频均匀抽帧 + 镜头边界抽帧 + 高运动区间加密抽帧）；
- 连续视频片段（供局部复核）；
- 镜头切分；
- OCR 文本（画面内文字逐条带时间戳）；
- ASR 音频转写；
- 音乐和环境音信息；
- 人物、产品、动作、场景变化清单；
- 前后帧一致性信号。

当前双模型（models.md）均支持原生视频理解，**直接整片输入，不做本地抽帧**——上述证据项由模型在原生看片时提取并带时间戳输出，keyframes/clips 仅在黄金集留档或人工复核时才生成。只有在被迫使用不支持原生视频的模型时，上述证据包才是最低要求（UVE-Bench 实验表明抽帧数量显著影响时序判断，稀疏抽帧会系统性损害运动、节奏、因果、一致性、音画同步五类评价）。

### 第二步：理解任务（Context Assembly）

Judge 的输入 = 任务上下文 + 生产约束 + 候选视频。有参照模式下评估时必须同时看到：

- 原始 Brief；
- 输入素材；
- 产品事实；
- 输出视频（证据包）；
- 平台要求；
- 品牌要求。

有参照模式缺任何一项即拒评；无参照模式跳过需参照的维度，**不允许 Judge 脑补约束**。输入契约见 `schemas/task_context.schema.json`。

**类型路由（两种模式都必跑）**：理解任务的第一件事是用 `prompts/genre_router.md` 判定视频类型、目标受众、观看场景、类型语言与该类观众的在意点/雷区。路由结果决定后续注入哪套类型 rubric 与观众人格——2026-07-15 试跑证明，拿口播带货的口径评 showcase 会把顶级作品系统性错杀（同一模型换对口径后 texture 从 1.5 分变 5.0 分，prompt 口径是主导变量）。路由 confidence 为 low 时用通用 rubric 并转人工确认类型。

### 第三步：建立质量表征（Quality Vector)

按 [dimensions.md](dimensions.md) 六层维度逐维打分并给出证据引用，形成 Q(y)。每个分数必须附至少一条可验证证据（时间戳/帧号/OCR-ASR 引文），没有证据的分数无效。

### 第四步：逼近人的最终选择（Preference Prediction）

Judge 真正预测的是 P(人类更偏好 A | x, c, A, B)——逼近目标人群的判断分布，而非发现绝对"视频真理"。校准方法见 [calibration.md](calibration.md)。

实现上分两条并行通道，**人格分离**：

- **专业诊断通道**（`prompts/dimension_judge.md` + 类型 rubric）：总监/质检员人格，按六层维度定位问题，回答"差在哪"，供修片和归因；
- **观众反应通道**（`prompts/audience_judge.md`）：观众人格（由类型路由注入 persona 与观看场景，高价值评估用目标用户/泛路人/同行 2–3 个人格），回答划走时刻、凭印象记住了什么、行为预测（赞/转/关注）、原话评价、完播档位——禁用专业术语，回答"值不值得看"。

两条通道不得互相污染：诊断通道发现的问题不自动扣"观看价值"，观众通道说"看完了很爽"也不豁免事实错误（Gate 与客观轨照常拦截）。**GSB 的"用户价值"判断以观众通道为锚，维度比较以诊断通道为据。**

## 二、第一层：致命错误门槛（Pointwise Gate）

先判断有没有不可接受的问题，任一命中即 Gate FAIL：

1. 产品或品牌错误；
2. 核心事实错误；
3. 关键内容缺失；
4. 严重人物、产品变形；
5. 音频缺失或语言错误；
6. 违反平台、法律或安全要求；
7. 文件不可播放；
8. 时长、比例、字幕等硬性要求不合格。

规则：一旦只有一方出现致命错误，整体 GSB 基本可直接确定（无致命方 ≥ G，除非它自己也接近不可用）。Gate 独立于 GSB 保留，因为 GSB 有基线依赖——基线很差时大量 G 不代表达到可交付标准。**Pointwise Gate 回答"能不能用"，Pairwise GSB 回答"有没有进步"，两套判断都要出。**

## 三、第二层：分维度比较

对 A/B 逐层比较，每层给出 win/tie/loss + 证据：

- Task Alignment：谁更符合 Brief；
- Visual Quality：谁的视觉质量更好；
- Temporal Quality：谁的时序表现更好；
- Audio Quality：谁的音频更好；
- Narrative：谁的叙事和节奏更好；
- Commercial Usability：谁更接近真实交付标准。

## 四、第三层：整体 GSB（决策层）

> 综合任务目标和用户价值，新方案相对基线是 G、S 还是 B？

- **G（Good）**：候选方案优于基线；
- **S（Same）**：两者没有实质差异；
- **B（Bad）**：候选方案劣于基线。

GSB = 决策层，不是完整评估体系。

### Same 的正确含义

> 两个视频虽然存在差别，但差别不足以产生稳定、明确、具有业务意义的偏好。

判 S 的典型情形：A 构图稍好但 B 节奏稍好；两者各有不同小瑕疵；Judge 无法形成高置信度判断；差异不足以让真实用户改变选择；改进幅度小于正常生成随机波动。

如果 GSB 不允许 Same，Judge 会被迫把随机差异解释为提升或退化，产生大量"虚假胜利"。S 是重要的**无差异区间**，报告时不得隐藏。

## 五、三个报告指标

以 G=42%、S=38%、B=20% 为例：

1. **G/S/B 分布**：完整披露三档比例，不隐藏 Same；
2. **净提升率** Net Lift = (G − B) / N = 22%，新方案带来的净正向偏好；
3. **排除 Same 后的胜率** Decisive Win Rate = G / (G + B) = 67.7%，在能明确分出胜负的样本中新方案赢了多少。

三项必须同时报告：若 95% 都是 Same，剩下几组样本的高 Decisive Win Rate 没有意义。

## 六、风险与控制

### 1. 位置偏差

- 随机化 A/B 顺序；
- 对重要样本交换位置再评一次；
- 换位后判断反转 → 标记为"不稳定"，降级为 S 或转人工；
- 不告诉 Judge 哪个是新模型、哪个是基线。

### 2. 单次判断噪声

高价值评估采用：多次采样；多 Judge 投票；交换位置复评；输出 G/S/B 概率而不是只有标签；低置信度样本转人工。

### 3. "更好看"替代"更好用"

Judge 最容易看到视觉质量，最容易忽略：产品信息正确性、品牌一致性、平台传播目标、素材使用要求、商业可交付性。所有评估 prompt 必须写明：**美学质量只是用户价值的一部分，不得自动覆盖任务完成度和事实错误。**

### 4. 看不见完整时间过程

优先原生视频理解模型；否则组合：全视频均匀抽帧 + 镜头边界抽帧 + 高运动区间抽帧 + ASR + 音频特征 + 局部视频片段复核。

### 5. 基线依赖

GSB 只回答"A 相对 B"。同时保留 Pointwise Gate（单条是否合格）与 Pairwise GSB（相对基线是否进步），两个口径都进报告。

## 七、单次评估运行的产物

每次运行在 `runs/<date>-<name>/` 留档：

```
runs/2026-07-15-pipeline-v3-vs-v2/
├── config.json          评的哪两批视频、评估模式、样本集、Judge prompt 版本、Judge 模型与采样设置
├── objective/           客观检查轨结果（规格、OCR/ASR 比对、响度等）
├── gate/                每条视频的 Gate 结果
├── dimensions/          每条视频的维度分 + 每对样本的分维度比较
├── gsb/                 每对样本的 GSB（含换位复评、概率、置信度）
└── report.md            G/S/B 分布 + Net Lift + Decisive Win Rate + 不稳定样本清单 + 转人工清单
```

Judge prompt 的版本演进记录在 `runs/judge-versions.md`（机制见 [evolution.md](evolution.md)）。
