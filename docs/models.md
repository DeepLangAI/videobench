# Judge 模型选型与已知坑

协议本身模型无关（docs/protocol.md 只要求每次 run 在 config.json 记录所用模型与 prompt 版本），但**当前生产配置采用双模型策略**，来自 2026-07 在 video-triage 流程中的实测总结。网关地址、API key、模型名统一配置在 `~/.claude/skills/video-triage/scripts/config.json`（本文件不复制 key），调用脚本为同目录 `analyze_video.py`。

## 双模型分工

| 模型 | 角色 | 覆盖维度 | 特点 |
|---|---|---|---|
| **gemini-3.5-flash** | 全维度主评 | L1–L6 全部，**音频类（L4）与音画配合的唯一信源** | 唯一能听到音频的模型（实测音频观察与字幕 VTT 逐字吻合）；抽帧 fps 默认 0.5 |
| **doubao-seed-2-0-pro-260215** | 视觉专项交叉验证 | L2 视觉 / L3 时序（`--preset visual`） | 只抽帧听不到音频，但视觉问题挖得最狠、时间戳最准；prompt 已禁止其猜测音频 |

两条命令**并行**跑（bash `&` + `wait`）：

```bash
python3 analyze_video.py --url <公网视频URL>                    # gemini 全维度
python3 analyze_video.py --url <公网视频URL> --preset visual    # doubao 视觉专项
```

本地文件用 `--file`（>30MB 建议传 URL）。复查局部：`--prompt "只看XX秒到XX秒，检查……" --fps 2 --model <模型名>`。

## 双模型结论合并规则

- 同一时间点**两个模型都报**的视觉问题 → 高置信；
- **仅一个模型报**的 → 标注来源，needs_human 确认；
- **音频和音画配合类问题只采信 gemini**（doubao 听不到）；
- 口播文本疑问用字幕 VTT 直接核对，VTT 是比两个模型都强的客观轨证据。

这天然构成 protocol.md 要求的"多 Judge 投票 + 证据配置多样性"（evolution.md 第 4 条）。

## 评分校准先验（重要）

实测（2026-07 三条片）：**模型直觉分普遍虚高 1–1.5 分**（模型给 3.5–4.7 的片子，人类实际打 2）。因此维度分不可直接采用模型的第一直觉，按确认后的问题密度锚定：

| 问题密度 | 锚定分 |
|---|---|
| 致命 ≥1 条 或 明显 ≥5 条 | 2 |
| 明显 3–4 条 | 3 |
| 明显 ≤2 条且无致命 | 4 |
| 基本无问题 | 5 |

模型直觉分只作参考附在证据里，最终分按密度锚定；黄金集校准（calibration.md）要持续监控这个虚高偏差是否仍存在。

## 已知坑（实测，避免重踩）

1. **qwen3.6-plus**：纯文本模型，看不了视频——不要用；
2. **qwen3-vl-flash**：贵（同片 125k tokens）且有幻觉——不要用；
3. **doubao 输入视频上限 50 MiB**，超限报 InvalidParameter——此时视觉维度以 gemini full 报告为准，并在报告注明该片缺 doubao 交叉验证（本机无 ffmpeg，压不了片）；
4. **gemini 知识纠错幻觉**：偶发拿自己的知识"纠正"画面文案（如把正确年份判成错别字）——时效性"错别字"类发现必须先核实；
5. **gemini 同音字听写不可靠**：实测把"第三夜"听成"第三页"、"说完"听成"说罢"——凡"读错字/念错稿"类发现，登记前必须用字幕 VTT 核对原稿；
6. **网关仅公司网络/特定 Wi-Fi 可达**：连不上先确认网络再排查；
7. 模型的问题清单是**草稿不是结论**：致命判定（Gate fail）必须有可验证证据，低置信转人工；
8. **gemini 对无音轨视频会整段幻觉音频**（2026-07-15 hyperframes-showcase 试跑实测，5/5 复现）：对没有音轨的视频，它会虚构 BGM 风格、"音乐戛然而止"、甚至完整的配音内容与 TTS 音色评价（把烧录字幕脑补成口播），并据此下"致命"判定。**任何音频类发现，必须先由客观轨确认音轨存在**（mdls 或 MP4 box 扫描），否则一律作废；
9. **video-triage 的内置 prompt 面向口播带货短视频**：评产品 showcase / kinetic typography 类片时，"弱钩子、无 CTA、语义断行"等标准系统性偏严（实测把逐屏歌词式文案全判成断行错误）。videobench 正式接入时需按视频类型分化 prompt。

## GSB Pairwise 的模型（2026-07-15 拍板）

GSB 沿用同一套双模型，不另选型：

- **gemini-3.5-flash** 做完整三层 GSB 判断（含音频层比较，它是唯一听得到音频的）；
- **doubao-seed-2-pro** 只做视觉/时序维度的成对比较，作为交叉验证；
- 合并：视觉/时序结论两模型一致 → 采纳；分歧 → 降置信度或转人工；音频/叙事口播类只采信 gemini；
- 换位复评、多次采样等偏差控制照常执行（protocol.md 第六节）。

待办：在黄金集上实测这套 GSB 配置与人类的一致率（UVE-Bench 显示成对比较对模型远难于单视频打分，先测再信）。

## 原生视频输入，不做本地抽帧

两个模型都支持原生视频理解（URL 或 base64 直传，模型侧自行采样），因此**评估不需要本地抽帧构造证据包**：

- `evidence.native_video_input` 恒为 true；keyframes/clips 字段仅在黄金集留档或人工复核时才填；
- `--fps` 是模型侧采样密度参数：默认 0.5 看结构，复查某几秒细节时对指定时间段提到 2；
- doubao 的"只抽帧"指它不解析音轨（听不到声音），不是要求本地抽帧。
